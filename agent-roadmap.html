Toy Agent Playground

Знакомство с базовыми концепциями coding agent: агентский цикл, роли сообщений, вызов инструментов и завершение задачи. Два источника модели: mock для тестов и один real provider (OpenAI-compatible) — пока захардкоженный.

Материалы: README Промпты DoD Edge cases

Зачем этот этап

  • Понять агент как цикл с явными состояниями, а не как «магию LLM».
  • Увидеть разницу между «модель думает» и «агент действует».
  • Заложить модель, которую все следующие этапы будут расширять, а не переписывать.

Чего здесь НЕ делаем

  • Нет переключения между несколькими провайдерами — один, захардкоженный.
  • Нет конфигурации через env / dotenv — ключ и URL прямо в коде (осознанный костыль).
  • Нет файловой системы, shell, git.
  • Нет web UI, persistence, approvals.
  • Нет отдельных библиотек — только минимальный прототип для обучения.

Концепции этапа

Agent Loopядро

Повторяющийся цикл: получить решение модели → выполнить действие → добавить результат в контекст → повторить. Имеет лимит шагов и явное завершение.

Conversation & Message Rolesконтекст

Диалог состоит из сообщений с ролями: пользователь задаёт задачу, ассистент принимает решения, инструмент возвращает результат действия. История сообщений — память агента на время run.

Tool Callдействие

Запрос на выполнение действия: имя инструмента + аргументы. Tool call — это данные, не side effect. Агент решает вызвать tool, runtime его исполняет и возвращает результат.

Final Answer vs Tool Callразвилка

На каждом шаге модель выбирает одно из двух: вызвать инструмент или дать финальный ответ пользователю. Это ключевое решение агентского цикла.

Mock Modelтесты

Подмена реальной LLM заранее заданными ответами. Детерминированные тесты, edge cases, работа без сети и API-ключей. Основной режим для pytest.

Real Provider (OpenAI-compatible)живой прогон

Один реальный провайдер с OpenAI-compatible API (OpenAI, OpenRouter, локальный vLLM/Ollama с совместимым endpoint). Пока захардкожены: base URL, API key, имя модели. Agent Loop не знает разницы — получает то же решение «tool call или final answer», что и от mock.

Mock vs Real — зачем обапрактика

Mock — для обучения и тестов. Real — чтобы увидеть, как модель сама выбирает tools на живой задаче. Цикл один и тот же; меняется только источник ответа модели. На этапе 2 это станет Protocol.

Run Lifecycleсостояние

Каждый run проходит через состояния: running → completed / failed / max_steps. Состояние должно быть явным и наблюдаемым.

Паттерны, которые закрепляем

  • State Machine — run имеет конечные состояния, переходы между ними предсказуемы.
  • Command — tool call описывает намерение, исполнение отделено от решения.
  • Explicit Errors — ошибка инструмента возвращается в контекст, а не роняет весь run.

Архитектура (концептуально)

User input → Agent Loop → Model ↔ Mock | Real → Tool call? → Tool → повтор → Final answer

Agent Loop держит историю сообщений и на каждом шаге спрашивает модель: «что дальше?» Источник ответа — mock (скрипт) или real provider (HTTP к OpenAI-compatible API). Модель либо просит вызвать инструмент, либо даёт финальный ответ. Результат инструмента добавляется в историю, и цикл продолжается.

Hardcode на этом этапе: base URL, API key и model name зашиты в коде real provider. Переключение mock / real — флагом при запуске (например, --mock vs --live). Это сознательно грубо: на этапе 2 вынесем за Protocol и конфиг.

Сценарий: happy path (mock)

Режим: mock model · Задача: «Сложи 2 и 3, потом скажи текущее время»

  1. Пользователь отправляет задачу → она становится первым сообщением в истории.
  2. Модель решает вызвать инструмент «сложение» с аргументами 2 и 3.
  3. Инструмент возвращает 5 → результат добавляется в историю.
  4. Модель видит результат и решает вызвать инструмент «текущее время».
  5. Инструмент возвращает timestamp → результат добавляется в историю.
  6. Модель формирует финальный ответ: «Sum is 5. Current time is …»
  7. Run завершается со статусом completed, потрачено 3 шага.

Сценарий: unknown tool

Задача: модель просит инструмент, которого нет.

  1. Модель запрашивает несуществующий инструмент.
  2. Runtime не падает — возвращает ошибку «unknown tool» в историю сообщений.
  3. Модель видит ошибку и может скорректировать поведение.
  4. Модель даёт финальный ответ: «Sorry, I don't have a tool for that.»
  5. Run завершается со статусом completed — ошибка tool не равна падению run.

Сценарий: max steps

Задача: модель зациклилась и бесконечно просит инструменты.

  1. На каждом шаге модель снова и снова вызывает один и тот же инструмент.
  2. Agent Loop считает шаги.
  3. При достижении лимита (например, 3) run принудительно останавливается.
  4. Статус: max_steps — это защита от бесконечного цикла.

Сценарий: real provider (live)

Режим: OpenAI-compatible provider, захардкоженный · Задача: «What is 17 + 25? Use the calculator tool.»

  1. Пользователь запускает агент с флагом live (не mock).
  2. Agent Loop отправляет историю сообщений + описание доступных tools в real API.
  3. Модель сама решает вызвать tool «add» с аргументами 17 и 25 — без скрипта.
  4. Runtime выполняет tool, результат 42 попадает в историю.
  5. Модель получает обновлённый контекст и формирует финальный ответ пользователю.
  6. Run завершается со статусом completed.

Отличие от mock: порядок и формулировки решает модель, ответы недетерминированы. Поэтому live-прогоны не входят в автотесты — только ручная проверка и демо ментору.

Сценарий: real provider — сеть недоступна

Режим: live · API не отвечает или ключ неверный.

  1. Agent Loop вызывает real provider → получает ошибку (timeout, 401, 429).
  2. Run завершается со статусом failed, ошибка видна пользователю.
  3. Run не зависает бесконечно — нужен timeout на запрос к API.

Джун понимает: mock покрывает логику цикла, real provider добавляет класс ошибок «внешний мир».

Что важно понять из примеров

  • Агент — это не один ответ модели, а цепочка шагов.
  • Каждый шаг добавляет информацию в контекст для следующего решения.
  • Ошибки инструментов — часть диалога, а не крах системы.
  • Лимит шагов — обязательная страховка, без неё агент может крутиться вечно.
  • Mock и real используют один и тот же Agent Loop — меняется только источник модели.
  • Live-прогоны недетерминированы; тесты всегда на mock.

этап 2 · 2 недели

Agent Core Library

Вынести агентский цикл из прототипа в переиспользуемое ядро. Mock и real provider — за одним контрактом ModelClient. Появляется поток событий, отмена run и конфиг вместо hardcode. CLI и web станут клиентами этого ядра.

Материалы: README Edge cases Все шаги

Зачем этот этап

  • Цикл из этапа 1 перестаёт жить внутри одного приложения — становится библиотекой.
  • Mock и real provider объединяются за одним контрактом, hardcode уходит.
  • Закладывается поток событий — фундамент для CLI, server и web UI.
  • Джун учится Dependency Inversion: ядро не зависит от деталей реализации.

Чего здесь НЕ делаем

  • Нет отдельного tool runtime — tools по-прежнему передаются снаружи (упрощённо).
  • Нет filesystem, shell, git, approvals.
  • Нет HTTP-сервера и web UI — только библиотека + тонкий клиент.
  • Нет streaming токенов от LLM — только события уровня шага/run.

Концепции этапа

Agent Coreбиблиотека

Переиспользуемый runtime: цикл, состояние run, события. Не знает, кто его вызывает — CLI, тест или server. Единственная ответственность: провести run от user input до финального статуса.

ModelClient (Protocol)порт

Контракт «дай историю сообщений → верни tool call или final answer». Mock и OpenAI-compatible реализуют один интерфейс. Agent Core зависит от Protocol, не от HTTP или скриптов.

Dependency Injectionсборка

ModelClient и tools передаются в агент при создании. Ядро не создаёт провайдер само — это делает клиент (toy_agent, позже CLI). Меняешь провайдер — ядро не трогаешь.

RunStateсостояние

Явная машина состояний run: pending → running → completed / failed / cancelled / max_steps. Любой наблюдатель может спросить «в каком состоянии run прямо сейчас».

AgentEvent & Event Streamнаблюдаемость

Каждый значимый момент run — событие: run.started, step.begin, tool.called, tool.result, run.completed. Поток событий — единый язык для CLI-рендера, логов и будущего web UI.

Cancellationконтроль

Run можно прервать извне (Ctrl+C, кнопка cancel). Ядро завершает текущий шаг gracefully и выставляет status cancelled — не обрывает процесс посреди записи в контекст.

Config вместо Hardcodeзрелость

API key, base URL, model name — из env или конфиг-файла, не из тела кода. Hardcode с этапа 1 осознанно снимается. Секреты не попадают в репозиторий.

Thin Clientграница

toy_agent после рефакторинга: только wiring (какой ModelClient, какой режим) + отображение событий. Вся логика цикла — в agent_core.

Паттерны, которые закрепляем

  • Ports & Adapters (Hexagonal) — ядро в центре, адаптеры (mock, OpenAI, CLI) снаружи.
  • Dependency Inversion — core → Protocol, не core → HTTP-клиент.
  • Observer / Pub-Sub — подписчики на AgentEvent, не опрос состояния.
  • State Machine — RunState с явными переходами.
  • Strategy — ModelClient как взаимозаменяемая стратегия.

Архитектура (концептуально)

Client → create_agent → Agent Core ↔ ModelClient → Event Stream → Observer

Клиент создаёт агент, передаёт ModelClient (mock или live) и набор tools, запускает run. Ядро крутит цикл и публикует события. Клиент подписывается и рисует вывод — или просто собирает события в лог. Тот же Agent Core позже будет стоять за HTTP API без изменений.

Связь с этапом 1

  • Agent Loop, Message Roles, Tool Call — без изменений по смыслу, меняется место жительства.
  • Mock model → реализация ModelClient.
  • Real provider → другая реализация того же ModelClient.
  • Hardcode → config.
  • Print в терминал → подписка на AgentEvent.

Сценарий: один run, два провайдера

Задача: та же user-задача, сначала mock, потом live — Agent Core и события идентичны по форме.

  1. Клиент создаёт агент с MockModelClient → запускает run.
  2. Поток событий: run.started → step.begin → tool.called → tool.result → … → run.completed.
  3. Клиент меняет только ModelClient на OpenAIModelClient (config из env) → тот же run API.
  4. Форма событий та же; меняется только содержимое (кто выбрал tool — скрипт vs модель).
  5. Джун видит: ядро не переписывалось, сменился адаптер.

Сценарий: отмена run

Задача: пользователь прерывает долгий run (Ctrl+C или cancel).

  1. Run в состоянии running, модель или tool на шаге N.
  2. Клиент сигнализирует отмену → ядро перестаёт запускать новые шаги.
  3. Текущий шаг по возможности завершается cleanly.
  4. Событие run.cancelled, финальный status: cancelled.
  5. История сообщений и событий до момента отмены сохранена — можно inspect.

Сценарий: replay событий

Задача: после run восстановить, что происходило, без повторного вызова LLM.

  1. Клиент сохранил список AgentEvent за run (в памяти или файл — не важно на этом этапе).
  2. Повторный прогон «рендера» по сохранённым событиям даёт тот же timeline.
  3. Это задел под web UI и server persistence на этапах 6–7.

Сценарий: ошибка ModelClient

Задача: live provider вернул 429 или timeout на шаге 2.

  1. ModelClient пробрасывает ошибку в ядро структурированно (не raw exception наружу).
  2. Событие run.failed с причиной.
  3. Status: failed. Run не зависает, не corrupts историю.

Что важно понять из примеров

  • Ядро стабильно — меняются адаптеры и клиенты, не цикл.
  • События — контракт между runtime и любым UI.
  • Отмена и ошибки провайдера — first-class состояния, не afterthought.
  • Config и секреты — ответственность клиента/адаптера, не ядра.

Неделя 1 — выделение ядра и контракт

Дни 1–2Границы
  • Что входит в agent_core, что остаётся в toy_agent.
  • Dependency rule: ядро не импортирует CLI, HTTP-библиотеки напрямую в цикле.
  • Перенос цикла из этапа 1 без изменения поведения.
Дни 3–4ModelClient
  • Protocol: один метод «complete» (или async-аналог).
  • MockModelClient — из mock этапа 1.
  • OpenAIModelClient — из live этапа 1, config из env.
  • Ядро работает с любой реализацией.
День 5RunState
  • Явные статусы и переходы.
  • Запрет невалидных переходов (completed → running).
  • RunState доступен наблюдателям через события и snapshot.

Неделя 2 — события, отмена, клиент

Дни 6–7AgentEvent
  • Каталог событий: run, step, tool, model, error.
  • Async-итератор или callback-подписка на поток.
  • toy_agent рисует timeline только из событий, не из print внутри цикла.
День 8Cancellation
  • Отмена через asyncio cancellation или явный cancel token.
  • Graceful shutdown: status cancelled, событие run.cancelled.
День 9Config
  • Env: API key, base URL, model name.
  • Нет секретов в git; .env.example для документации.
День 10Закрепление
  • Тесты ядра только с FakeModelClient — без network.
  • Ручной live-прогон через toy_agent.
  • Устное объяснение: зачем события понадобятся на этапе 6 (server).
